## 问_1
在C++11标准中，不同的锁有什么不同，比如lock_guard和unique_lock

## 答_1
在 C++11 标准中，`std::lock_guard` 和 `std::unique_lock` 是最常用的两种 RAII（资源获取即初始化）锁管理工具。它们的主要区别在于**灵活性**和**性能开销**。

以下是它们的详细对比：

### 1. `std::lock_guard` (轻量级，简单)
它是最基础的 RAII 包装器，在构造时加锁，析构时解锁。
- **特点**：
  - **简单高效**：代码路径极短，几乎没有额外开销。
  - **不可手动控制**：一旦创建，你就不能手动解锁（unlock）或再次加锁（lock）。它必须等到作用域结束（析构）才会释放锁。
  - **不可转移**：不能被移动（move）或复制。
- **适用场景**：当你只需要在某个代码块（如函数体、循环体）全程持有锁时，首选 `lock_guard`。

```cpp
void func() {
    std::lock_guard<std::mutex> lock(mtx);
    // 临界区代码...
} // 自动解锁
```

### 2. `std::unique_lock` (功能丰富，灵活)
它是 `lock_guard` 的增强版，提供了更多的控制选项。
- **特点**：
  - **手动开关**：可以在生命周期内随时调用 `.unlock()` 和 `.lock()`。这对于减少锁的粒度非常有用。
  - **延迟加锁**：构造时可以传入 `std::defer_lock`，表示先关联互斥量但不立即加锁。
  - **可转移**：支持移动语义（`std::move`），可以将锁的所有权从一个函数转移到另一个函数，或者存入容器。
  - **配合条件变量**：使用 `std::condition_variable` 时，**必须**配合 `unique_lock` 使用，因为条件变量在等待时需要频繁地释放和重新获取锁。
  - **开销稍大**：需要维护一个内部状态来记录当前是否持有锁。
- **适用场景**：需要配合条件变量、需要中途解锁、或者需要转移锁所有权的复杂场景。

```cpp
void func() {
    std::unique_lock<std::mutex> lock(mtx);
    // 临界区代码...
    lock.unlock(); // 手动提前释放
    // 做一些耗时但不需锁的操作...
    lock.lock();   // 重新获取
}
```

### 3. 核心差异总结表

| 特性 | `std::lock_guard` | `std::unique_lock` |
| :--- | :--- | :--- |
| **灵活性** | 低 (构造加锁，析构解锁) | 高 (可中途加锁/解锁) |
| **性能** | 极高性能 (开销最小) | 略有开销 (需维护状态) |
| **条件变量** | 不支持 | **必须使用它** |
| **所有权转移** | 不支持 | 支持 (可移动) |
| **尝试加锁 (try_lock)** | 不支持 | 支持 |

### 针对你当前项目的建议
在你现在的线程池项目（如 ThreadPool.cpp）中：
1. **任务队列的 `push` 操作**：如果只是简单的入队，使用 `std::lock_guard` 即可。
2. **工作线程等待任务**：由于需要配合 `std::condition_variable::wait()`，你**必须**在工作线程中使用 `std::unique_lock`。
