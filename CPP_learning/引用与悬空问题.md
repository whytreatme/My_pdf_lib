# 描述  
本文是关于C++语法的引用与局部变量的问题

## 问 
> “为什么按值传入会导致引用悬空？引用不是能“延长”临时对象生命周期吗？
关键点：“生命周期延长”只发生在临时对象直接绑定到引用的那一刻，而且只对那个引用有效。”

### 问题描述  
还有就是，什么式临时对象直接绑定引用，什么式局部对象被销毁，如果我在主函数中有以下代码：
int& a;
{
    int b = 5;
    a = b;
}
那b也是局部变量，那a悬空了吗？请你也有代码给我演示

### 回答   
先纠正一下你举的例子：  
`int& a;` 这种写法在 C++ 里**不合法**，引用必须在声明时就初始化。

下面我用可编译的代码演示三种情况，分别说明“临时对象绑定引用”和“局部对象销毁导致悬空”是什么。

---

**情况 A：引用绑定到局部变量（会悬空）**
```cpp
int& badRef() {
    int b = 5;     // b 是局部变量
    return b;      // 返回 b 的引用
} // b 在这里被销毁

int main() {
    int& a = badRef();   // a 绑定到已经销毁的 b
    // 这里 a 是悬空引用，使用它是未定义行为
}
```
这就是“局部对象被销毁，引用悬空”。

---

**情况 B：引用绑定到临时对象（生命周期延长，仅限绑定的那个引用）**
```cpp
int main() {
    const int& a = 5;  // 临时对象 5 的生命周期延长到 a 的作用域结束
    // 这里 a 是安全的
}
```
这种“生命周期延长”只发生在**临时对象直接绑定到引用**时，而且通常是 `const T&` 或 `T&&`。

---

**情况 C：你想表达的“外部引用指向局部变量”的效果**
如果硬要模拟你想表达的那段逻辑，可以写成下面这样：

```cpp
int main() {
    int* p = nullptr;
    {
        int b = 5;
        p = &b;     // 保存 b 的地址
    } // b 在这里销毁
    // *p 现在是悬空指针
}
```

这段代码和你想表达的“把外层引用指向内层局部变量”是同一种问题：  
局部变量生命周期结束后，外部还指着它，就悬空了。

---

回到你最初的疑问：  
你如果在构造函数里 **按值传入** `EventLoop loop`，`loop_` 引用绑定的其实是**那个形参对象**（局部变量），构造函数结束时它就被销毁，所以 `loop_` 变悬空。

而你现在已经改成 `EventLoop& loop` 作为参数，这就不会发生这个问题。
