### 一、 程序员的密语：Conventional Commits

这些缩写是 **约定式提交 (Conventional Commits)** 规范，它让全球开发者看到你的日志就能瞬间明白你做了什么。

*   **`feat` (feature)**：**新功能**。你增加了转账功能，或者增加了管理员控制台。
*   **`fix` (bug fix)**：**修复错误**。修复了一个内存泄露，或者处理了一个 SQL 注入漏洞。
*   **`perf` (performance)**：**性能优化**。这就是你现在做的：引入线程池，提高并发处理能力。
*   **`refactor`**：**重构**。代码逻辑没变，但你把乱糟糟的代码整理得更漂亮了。
*   **`docs`**：**文档**。你改了 README 或者增加了注释。
*   **`chore` (琐事)**：**日常维护**。改了改编译脚本（CMake/Pro文件），或者清理了几个没用的临时文件。
*   **`style`**：**样式修改**。只是改了缩进、加了空格，没动逻辑。

**为什么要这么写？**
当你的项目变大时，自动化工具可以根据这些前缀自动生成“版本更新日志（Changelog）”。

---

### 二、 README.md：它不是静态的，它是“进化史”

你担心“现在写了单线程，改了多线程怎么办？”——**这正是展示你实力的机会。**

README 不仅描述现状，还可以描述**架构演进 (Evolution)**。你可以这样写：

```markdown
## 系统架构

### V1.0 - 基础版 (Single-Threaded Reactor)
- 基于 Qt 事件循环的单线程处理模型。
- **局限性**：在高并发场景下（>100连接），数据库 I/O 阻塞主线程，导致连接拒绝。

### V2.0 - 高性能版 (Multi-Threaded Worker Pool) [当前分支]
- 引入 I/O 与 业务逻辑分离。
- **主线程**：专职处理套接字读写（Reactor）。
- **工作线程池**：通过 `QThreadPool` 处理数据库查询与逻辑计算。
- **并发表现**：支持 500+ 并发连接，错误率为 0%。
```
**这种写法在求职面试或项目展示时杀伤力极大**，因为它证明了你不仅会写代码，还具备发现问题并进行架构重构的能力。

### 三、 Design Document (设计文档)：灵魂的解剖

你提到的 Design Doc 是资深开发者的必修课，在 Google、Amazon 等公司被称为 **Design Doc** 或 **RFC (Request for Comments)**。

它不写在代码里，通常写在一个独立的 Markdown 文件中（比如 `docs/architecture_v2_pool.md`）。它的结构如下：

#### 1. Objective (目标)
一句话说明你要做什么：*“引入线程池模型，将服务器并发处理能力从单线程 50 次/秒 提升至 500 次/秒。”*

#### 2. Background & Motivation (背景与动机)
描述痛点。*“由于 MySQL 查询是同步阻塞的，当前单线程模型在 500 并发压测下出现 40% 的 Connection Refused。这是因为主线程被 I/O 锁死，无法及时调用 accept()。”*

#### 3. Proposed Design (提议设计 - 核心部分)
*   **架构图**：用简单的文字或 ASCII 画出流量流向：
    `Client -> Reactor(Main Thread) -> WorkerTask(ThreadPool) -> DB`
*   **关键变动**：
    *   `TcpReactor` 不再直接调用 Controller。
    *   实现 `WorkerTask` 类继承自 `QRunnable`。
    *   **数据库连接池策略**：描述你如何通过线程 ID 隔离 MySQL 连接（这是你这个设计的“高级感”来源）。

#### 4. Trade-offs (权衡与备选方案)
这是体现你思考深度的地方。
*   **方案 A (弃用)**：增加单线程连接数。*理由：无法解决 I/O 阻塞根本问题。*
*   **方案 B (采用)**：线程池。*优势：吞吐量极高。挑战：需要处理跨线程数据库连接。*

#### 5. Impact (影响)
改动后会影响哪些模块？*“DAO 层需要全面适配 `getThreadConnection()` 方法。”*

### 四、 实际开发人员是怎么做的？（工程节奏）

如果你在一个正规的技术团队，流程通常是这样的：

1.  **需求分析**：发现压测失败。
2.  **写设计稿 (Design Doc)**：在团队 Wiki 上发帖，描述你打算怎么改。
3.  **评审 (Review)**：架构师（也就是我）会看你的方案，指出：“嘿，你要注意 MySQL 的最大连接数限制”。
4.  **开发分支**：创建 `feature/thread-pool` 分支。
5.  **代码评审 (Code Review)**：合并回主分支前，大家一起挑刺。
6.  **部署与验证**：重新跑压力测试脚本，用数据说话。
