### 1. 架构师视角：什么是“设计”？

在软件工程中，设计分为两个层面：

*   **宏观设计（Architectural Design）**：
    *   我们要解决高并发崩溃。
    *   我们要把 I/O 和业务逻辑分离（方案三）。
    *   我们要解决数据库连接的线程安全性。
    *   我们要通过线程 ID 隔离连接名。
    *   **这些决策是谁做的？是你。** Copilot 并没有告诉你“你应该用方案三”，是你根据我的建议，结合你的理解，选择了这条路。这就是设计。

*   **微观实现（Implementation Detail）**：
    *   Qt 的线程 ID 怎么拿？（`QThread::currentThreadId()`）
    *   怎么判断连接是否存在？（`QSqlDatabase::contains()`）
    *   **这些琐碎的 API 调用，不叫设计，叫“搬砖”。** 即使没有 Copilot，你也只是在查文档和 Google。把大脑花在记 API 上，才是真正的浪费。

### 2. 如何利用 Copilot 进行“刻意练习”？

如果你担心自己不动脑子，我教你一套**“反向学习法”**，能让你在 Copilot 的辅助下，技术比别人提升快 5 倍：

#### 第一步：先“盲猜”，再“提问”
在问 Copilot 之前，先拿一张纸或在注释里写下你的**伪代码逻辑**。
*   *比如*：我觉得应该先拿线程名，然后 `if` 判断有没有，没有就 `addDatabase`，最后 `setHost`。
*   **然后再问 Copilot**。如果它的代码和你写的一样，说明你掌握了逻辑；如果它多写了几行（比如处理了 `m_config`），你要立刻追问：**“为什么你要加这一行？不加会怎么样？”**

#### 第二步：手动录入，拒绝 Tab
这是最笨但最有效的办法：**看着 Copilot 给出的代码，自己一行行敲进去。**
*   当你亲手敲下 `QSqlDatabase::addDatabase(type, name)` 时，你的手指会产生肌肉记忆，你的大脑会强迫你思考这两个参数的关系。
*   **绝对不要直接按 Tab 键全盘接受。**

#### 第三步：进行“破坏性实验”
AI 给出的代码通常是“标准答案”。你可以故意改坏它。
*   *实验*：如果不给 `addDatabase` 传第二个参数（连接名）会怎样？
*   *结果*：你会发现所有线程都在抢“默认连接”，程序依然崩溃。
*   **这种“对比实验”带来的技术深度，是看书和问 AI 都给不了你的。**
