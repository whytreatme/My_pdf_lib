# 前提    
```cpp
/*
* 我的回显服务器
*/
#include <iostream>
#include <csignal>
#include <thread>
#include <pthread.h>
#include "EchoServer.h"
using namespace std;

int main(int argc, char *argv[]){
    if(argc != 5){
        cout << "Usage " << argv[0] << " ip port timerIntervalSec idleTimeoutSec\n";
        cout << "./main 0.0.0.0 5005 60 180\n\n";
        return -1;
    }

    // 1) 阻塞信号，交给 sigwait 处理
    sigset_t set;
    sigemptyset(&set);
    sigaddset(&set, SIGINT);
    sigaddset(&set, SIGTERM);
    pthread_sigmask(SIG_BLOCK, &set, nullptr);

   
     EchoServer *echoserver = new EchoServer(argv[1], atoi(argv[2]), 3, 5, atoi(argv[3]), atoi(argv[4]));
   // 2) 让服务器在独立线程运行
    std::thread serverThread([&]{
        int sig = 0;
        sigwait(&set, &sig);
        echoserver->Stop();
    });
    echoserver->Start();

   
    serverThread.join();
    delete echoserver;

    return 0;
}
```

## 问_1     
你给我解释解释什么是异步信号安全，这份代码我看不懂   

## 答_1  
**异步信号安全（Async-Signal-Safety）** 是多线程和系统编程中非常重要的概念。简单来说，如果一个函数可以在信号处理函数（Signal Handler）中被安全地调用，它就是“异步信号安全”的。

为了让你理解 main.cpp 里的代码，我们先拆解这个概念，再看代码是怎么规避风险的。

---

### 1. 什么是异步信号安全？

当程序运行到一半时，内核可能会随时触发一个信号（如 `SIGINT` 快捷键 Ctrl+C）。此时，内核会**强行中断**主线程的执行流程，临时去执行信号处理函数。

**风险点：导致死锁或冲突**
假设主线程正在执行 `malloc`（它内部有一把全局锁），如果主线程还没释放锁就被中断，而信号处理函数里也调用了 `malloc`，那信号处理函数就会陷入永久等待，造成**死锁**。

**结论：**
*   **异步信号安全函数**：在信号中断期间再次进入（Re-entrant）也不会产生副作用的函数（如 `write`, `read`, `_exit`, `sigwait`）。
*   **非异步信号安全函数**：使用了全局变量、静态变量或锁的函数（如 `printf`, `malloc`, `free`）。**绝对不应该在普通的信号处理函数中调用这些函数。**

---

### 2. 代码解析：它是如何处理信号的？

你提供的 main.cpp 并没有使用传统的 `signal()` 或 `sigaction()` 注册回调函数，而是采用了更高级、更安全的**同步化处理信号**方案。

#### 第一步：阻塞信号
```cpp
sigset_t set;
sigemptyset(&set);
sigaddset(&set, SIGINT);  // Ctrl+C
sigaddset(&set, SIGTERM); // 终止信号
pthread_sigmask(SIG_BLOCK, &set, nullptr); // 阻塞这些信号，不让它们中断当前逻辑
```
这部分代码告诉系统：“先别急着中断我，我会自己找时间处理这些信号。”

#### 第二步：开启独立线程同步处理
```cpp
std::thread serverThread([&]{
    int sig = 0;
    sigwait(&set, &sig); // 关键：在这里“坐等”信号发生
    echoserver->Stop();  // 这里的 Stop() 是安全的！
});
```
*   **`sigwait`**：它会将线程挂起，直到 `set` 中的信号发生。因为它是同步等待，**不是异步中断**，所以它不会破坏主程序的执行流。
*   **核心优势**：在 `serverThread` 线程中，当 `sigwait` 返回时，它就像执行普通程序逻辑一样。你可以在这里安全地调用 `echoserver->Stop()`，甚至在里面用 `printf` 或 `delete` 对象，而**完全不需要担心异步信号安全问题**。

---

### 3. 为什么要这么写？

在高性能网络服务器（如你的 `muduo_Reactor`）中，直接写一个 `void handler(int sig)` 函数是很危险的，因为：
1.  你不能在 `handler` 里正常关闭服务器（关闭 Socket、释放内存、打印日志都不是异步安全的）。
2.  通过 `sigwait` 方案，你把“异步”的信号转变成了“同步”的控制流程。

**总结：**
这份代码通过 **“先阻塞信号，再用独立线程 `sigwait` 同步处理”** 的模式，避开了“函数必须是异步信号安全”的严苛限制，让服务器的退出逻辑（`Stop()`）可以像编写普通代码一样自由、安全。


## 问_2    
那这里面的sig起到什么的作用   

## 答_2  
在 `sigwait(&set, &sig)` 中，那个变量 `sig` 的作用非常简单：**它是用来接收“到底是谁触发了信号”的。**

具体解释如下：

### 1. 信号的“收件箱”
`sigwait` 函数的第二个参数是一个整数指针。当程序因为收到信号从 `sigwait` 阻塞状态唤醒时，内核会将触发唤醒的**信号编号**（比如 `SIGINT` 的值是 2，`SIGTERM` 的值是 15）写入到这个 `sig` 变量中。

### 2. 代码中的实际用途
在 main.cpp 里的逻辑是：
```cpp
int sig = 0;
sigwait(&set, &sig); // 程序运行到这里会“躺平”，直到收到 SIGINT 或 SIGTERM
echoserver->Stop();  // 一旦 sigwait 填入信号值并返回，就开始停止服务器
```
虽然在这段代码中，作者并没有去检查 `sig` 的具体数值（只要是 `set` 里的信号，都执行 `Stop()`），但在更复杂的场景下，它是很有用的：

*   **区分处理**：如果你想收到 `SIGINT` 时只打印日志，收到 `SIGTERM` 时才彻底关闭服务器，你可以这么写：
    ```cpp
    sigwait(&set, &sig);
    if (sig == SIGINT) {
        // 处理 Ctrl+C 的逻辑
    } else if (sig == SIGTERM) {
        // 处理终止命令的逻辑
    }
    ```

### 3. 为什么必须传它？
根据 POSIX 标准，`sigwait` 的 API 定义强制要求传入一个 `int *` 来存储信号值。即使你代码里不打算用它（像这份代码一样），你也必须声明一个 `int` 并把它的地址传进去，否则编译器会报错。

**总结：**
`sig` 在这里起到一个**出参（Output Parameter）**的作用，告诉线程：“由于有人发了信号（编号存在我这了），你该起床干活（执行 Stop）了。”
