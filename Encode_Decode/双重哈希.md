## 例子  
```cpp  
QString ProtocolUtils::hmacSha256(const QByteArray& data, const QByteArray& key)
{
    const int blockSize = 64;  // SHA-256 块大小

    // 1. 准备密钥
    QByteArray keyBlock = key;
    if (keyBlock.size() > blockSize) {
        keyBlock = QCryptographicHash::hash(key, QCryptographicHash::Sha256);
    }
    while (keyBlock.size() < blockSize) {
        keyBlock.append('\0');
    }

    // 2. 生成 ipad 和 opad
    QByteArray iPadded;
    QByteArray oPadded;
    for (int i = 0; i < blockSize; ++i) {
        iPadded.append(static_cast<char>(keyBlock[i] ^ 0x36));  // ipad = 0x36
        oPadded.append(static_cast<char>(keyBlock[i] ^ 0x5c));  // opad = 0x5c
    }

    // 3. 计算内层哈希: H(K ^ ipad || data)
    QByteArray innerHash = QCryptographicHash::hash(
        iPadded + data,
        QCryptographicHash::Sha256
    );

    // 4. 计算外层哈希: H(K ^ opad || innerHash)
    QByteArray outerHash = QCryptographicHash::hash(
        oPadded + innerHash,
        QCryptographicHash::Sha256
    );

    // 5. 返回前 15 字节的十六进制字符串（30 个字符）
    QByteArray shortenedHash = outerHash.left(15);
    return QString::fromUtf8(shortenedHash.toHex());
}
```   

### 制作两个特殊的“垫板” (ipad 和 opad)  

```cpp  
// ...
// 2. 生成 ipad 和 opad
QByteArray iPadded;
QByteArray oPadded;
for (int i = 0; i < blockSize; ++i) {
    iPadded.append(static_cast<char>(keyBlock[i] ^ 0x36));  // ipad = 0x36
    oPadded.append(static_cast<char>(keyBlock[i] ^ 0x5c));  // opad = 0x5c
}
```  
ipad (inner pad) 和 opad (outer pad) 是两个固定的、像“魔法数字”一样的值。
^ 是“异或”运算，是一种位运算。你可以简单理解为，它把我们的密钥 keyBlock 和这两个“魔法数字”分别进行混合，生成了两个新的、独一无二的“垫板”：iPadded 和 oPadded。
目的：从一个密钥派生出两个用于不同阶段的密钥，一个用于“内部”，一个用于“外部”。   

### 计算内层哈希 (第一次哈希)  
```cpp  
// ...
// 3. 计算内层哈希: H(K ^ ipad || data)
QByteArray innerHash = QCryptographicHash::hash(
    iPadded + data,
    QCryptographicHash::Sha256
);
```   
iPadded + data：把你的原始数据 (data) 放在“内部垫板”(iPadded) 上。
QCryptographicHash::hash(...)：对这个组合体做一次标准的 SHA-256 哈希。
结果：得到一个“内层”的哈希结果 innerHash。这相当于把信件和第一层保护膜一起封进了信封。  

### 计算外层哈希 (第二次哈希)  
```cpp  
// ...
// 4. 计算外层哈希: H(K ^ opad || innerHash)
QByteArray outerHash = QCryptographicHash::hash(
    oPadded + innerHash,
    QCryptographicHash::Sha256
);
// ...  
```  
oPadded + innerHash：现在，把上一步得到的“内层哈希结果”(innerHash)，放在“外部垫板”(oPadded)上。
QCryptographicHash::hash(...)：对这个新的组合体再做一次 SHA-256 哈希。
结果：得到最终的、非常安全的签名 outerHash。这相当于在第一个信封外面，再套上一个用第二层保护膜加固的信封。  

### 双重哈希的作用  
我们把哈希想象成一个无法逆转的果汁机：

第一次榨汁 (计算 innerHash)：

你把水果 (data) 和第一种秘密调料 (iPadded，它来自你的密钥) 一起扔进果汁机。
你得到了一杯混合果汁 (innerHash)。
关键点：现在你只看到了这杯果汁，你已经无法从果汁里把完整的水果和调料分出来了。这杯果汁既包含了水果的信息，也包含了秘密调料的信息。
第二次榨汁 (计算 outerHash)：

现在，你把刚才那杯混合果汁 (innerHash) 和第二种秘密调料 (oPadded，也来自你的密钥) 一起扔进果汁机，再榨一次。
你得到了最终的“超级混合果汁” (outerHash)。
为什么不能跳过第一步？

如果第二次榨汁时，你还是用原来的水果 (data) 和第二种秘密调料 (oPadded)，那两次榨汁过程就相互独立了。攻击者可能会找到方法，分别攻击这两个独立的过程。

而 HMAC 的设计，让第二次哈希的输入 (innerHash) 依赖于 第一次哈希的结果。这意味着：

innerHash 就像一个中间产物，它把原始数据和密钥的影响传递给了下一步。

这个“传递”过程是 HMAC 算法安全性的核心。它确保了最终的签名结果 (outerHash) 同时受到了 data、iPadded 和 oPadded 三者的共同影响，把它们“纠缠”在了一起，让攻击者无法从任何一个环节找到突破口。
